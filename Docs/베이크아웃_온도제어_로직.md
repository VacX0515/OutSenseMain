# VacX OutSense — 베이크아웃 온도 램프업 및 유지 로직

## 시스템 개요

**VacX OutSense**는 진공 챔버의 탈가스율 측정 및 베이크아웃을 자동화하는 C# WinForms 애플리케이션입니다.

### 하드웨어 구성

| 장비 | 모델 | 역할 | 통신 |
|------|------|------|------|
| **온도 컨트롤러** | Autonics TM4-N2SE | 히터 PID 제어, 온도 측정 | Modbus RTU, 19200/8/N/1 |
| **확장 모듈 1** | TM4-N2SE (슬레이브 2) | 추가 온도 센서 4채널 | 동일 COM 포트 |
| **확장 모듈 2** | TM4-N2SE (슬레이브 3) | 추가 온도 센서 4채널 | 동일 COM 포트 |
| **칠러** | Lab Companion LK-1000 | 냉각수 순환 온도 제어 | Modbus RTU |
| **터보 분자 펌프** | Pfeiffer HiPace | 고진공 배기 | USS 프로토콜 |
| **드라이 펌프** | Edwards nXDS | 저진공 배기 | 시리얼 |
| **이온 게이지** | PTR225 | 고진공 압력 측정 | 아날로그 전압 |
| **I/O 모듈** | M31-XAXA0404G | 밸브 제어, 디지털/아날로그 입출력 | Modbus RTU |

### 온도 채널 구성 (총 12채널)

```
메인 TM4 (슬레이브 1):  Ch1(히터), Ch2, Ch3, Ch4     ← 메인 모듈
확장 TM4 (슬레이브 2):  Ch5, Ch6, Ch7, Ch8           ← 확장 모듈 1
확장 TM4 (슬레이브 3):  Ch9, Ch10, Ch11, Ch12        ← 확장 모듈 2
```

- **Ch1**: 히터에 직접 부착된 센서 (히터 온도 = TM4 내장 PID의 제어 대상)
- **Ch2~Ch12**: 샘플/챔버 각 위치에 부착된 센서 (모니터링 또는 PI 피드백 기준)

### TM4 컨트롤러 주요 기능

TM4는 독립적으로 동작하는 하드웨어 PID 온도 컨트롤러입니다:

- **내장 PID**: SV(설정값)를 추종하여 히터 출력(MV)을 자율적으로 제어
- **하드웨어 램프**: SV를 지정 속도(°C/h)로 자동 상승/하강 (통신 없이도 동작)
- **Modbus 인터페이스**: PC에서 SV 설정, RUN/STOP, 램프 속도 변경, PV/MV 읽기 가능

> TM4의 PID와 소프트웨어 PI는 별개입니다. TM4 내장 PID는 항상 동작하며 SV→MV를 제어하고, 소프트웨어 PI는 샘플 온도를 기준으로 SV 자체를 조정하는 **캐스케이드 제어** 구조입니다.

## 베이크아웃 온도 제어

### 두 가지 제어 모드

베이크아웃 온도 제어는 **모니터 채널 선택**에 따라 두 가지 모드로 동작합니다.

| 모드 | 모니터 채널 | 승온 제어 | 온도 유지 | 제어 대상 |
|------|------------|----------|----------|----------|
| **TM4 직접 제어** | Ch1만 선택 | TM4 하드웨어 램프 | TM4 내장 PID | 히터 온도 |
| **소프트웨어 PI 제어** | Ch2~Ch12 중 선택 | 소프트웨어 램프 + PI 피드백 | PI 피드백 유지 | 샘플 온도 |

---

## 1. TM4 직접 제어 모드 (Ch1만 모니터)

### 동작 원리

Ch1은 히터에 직접 부착된 센서이므로, TM4 컨트롤러의 **내장 PID**가 직접 온도를 제어할 수 있습니다. PC 소프트웨어는 초기 설정만 하고 이후 개입하지 않습니다.

```
┌─────────────────────────────────────────────┐
│                 TM4 컨트롤러                  │
│                                              │
│  [하드웨어 램프]                               │
│    SV를 BakeoutRampRate(°C/h)로 자동 상승      │
│           │                                   │
│           ▼                                   │
│  [내장 PID]  SV → 히터 출력(MV) 제어           │
│           │                                   │
│           ▼                                   │
│  [히터] → [Ch1 센서] → PV                     │
└─────────────────────────────────────────────┘

PC 소프트웨어: 목표 도달 감시 + 안전 체크만 수행 (제어 개입 없음)
```

### 시퀀스

1. **램프 설정**: TM4의 램프 레지스터에 `BakeoutRampRate` (°C/h) 기록
2. **목표 온도 설정**: TM4의 SV 레지스터에 최종 목표온도 기록
3. **히터 시작**: TM4 RUN 코일 활성화
4. TM4가 자체적으로 SV를 램프 속도에 맞춰 서서히 올림
5. SV가 목표온도에 도달하면 TM4 내장 PID가 온도 유지
6. 소프트웨어는 5초 주기로 PV를 읽어 목표 도달 여부와 안전 상태를 감시

### 장점

- TM4 내장 PID의 정밀한 히터 제어 (오토튜닝 지원)
- PC 통신 지연/끊김에 영향 없음 (하드웨어 자율 동작)
- 단순한 구조

### 제한

- 히터(Ch1) 온도만 제어 가능 — 샘플 온도와 히터 온도 사이에 열 지연 존재
- 샘플 온도 오버슈트를 직접 보상할 수 없음

## 2. 소프트웨어 PI 제어 모드 (Ch2~Ch12 포함)

### 동작 원리

샘플에 부착된 센서(Ch2~Ch12)의 온도를 기준으로, PC 소프트웨어가 Ch1(히터)의 SV를 실시간으로 조정합니다. 이것은 **캐스케이드 제어** 구조입니다:

- **외부 루프 (소프트웨어 PI)**: 샘플 온도 → Ch1 SV 결정
- **내부 루프 (TM4 내장 PID)**: Ch1 SV → 히터 출력(MV) 제어

```
┌──────────────────────────────────────────────────────────────┐
│  PC 소프트웨어 (5초 주기)                                      │
│                                                               │
│  [샘플 센서 Ch2~12]                                           │
│    모니터 온도(MAX) ─────→ [소프트웨어 PI] ────→ Ch1 SV 설정    │
│         ↑                       │                              │
│         └───────────────────────┘                              │
│                           피드백 루프                           │
└──────────────────────────────────────────────────────────────┘
                                │
                                ▼  Modbus 쓰기 (SV 레지스터)
┌──────────────────────────────────────────────────────────────┐
│  TM4 컨트롤러                                                 │
│  [내장 PID]  SV → 히터 출력(MV) 제어                           │
└──────────────────────────────────────────────────────────────┘
```

> 다중 모니터 채널이 선택된 경우, 각 채널 온도의 **최대값(MAX)**을 기준으로 제어합니다. 이는 샘플의 가장 뜨거운 지점이 목표를 초과하지 않도록 보장합니다.

### 승온 2단계: 학습 → 리딩

PI 모드의 승온은 **학습 단계**와 **리딩 단계**로 나뉩니다. 초기에는 챔버/샘플의 열 특성을 알 수 없으므로 보수적으로 시작하고, 열응답이 파악된 후 본격적으로 제어합니다.

```
rampedTarget = 초기샘플온도 + (경과시간_초 × BakeoutRampRate / 3600)
rampedTarget = min(rampedTarget, 최종목표온도)
```

> TM4 하드웨어 램프는 **비활성화** (램프속도=0으로 설정). PI가 Ch1 SV를 직접 덮어쓰기 때문에 하드웨어 램프와 충돌합니다.

#### Phase 1: 학습 (Learning)

히터를 시작하면서 열 응답을 관측합니다.

```
초기 CH1 SV = 샘플온도 + 3°C (작은 오프셋에서 시작)

매 5초마다:
  currentLag = CH1_PV - 샘플온도
  observedThermalLag = EMA(observedThermalLag, currentLag, α=0.2)  // 빠른 EMA
  CH1 SV = rampedTarget + observedThermalLag + 3°C                 // 보수적 추종

학습 완료 조건 (모두 충족):
  - 최소 6회(30초) 관측
  - 열지연 변화량 < 0.3°C/사이클이 4회(20초) 연속
  - observedThermalLag > 0.5°C (유의미한 열지연 관측)
```

학습 중에는 PI 적분을 누적하지 않고, CH1 SV를 `rampedTarget + 관측 열지연` 수준으로만 설정합니다. 이 기간 동안 샘플 온도는 CH1의 느린 열 전달에 의해 자연스럽게 상승합니다.

> CH1이 이미 가열된 상태에서 재시작하는 경우 (중단 후 이어하기), 기존 열지연을 즉시 파악할 수 있으므로 학습을 건너뛰고 리딩으로 직접 전환합니다.

#### Phase 2: 리딩 (Leading)

학습 완료 후, 관측된 열지연을 적분 초기값으로 설정하고 본격적인 PI 제어를 시작합니다.

```
// 학습→리딩 전환 시
integralTerm = observedThermalLag × 0.85  // 정상상태 열지연 보상용 적분 초기값

// 이후 정상 PI 제어
error = rampedTarget - 모니터온도(MAX)

// 변화율 (이동평균 필터링, α=0.3)
smoothedRate = smoothedRate × 0.7 + rawRate × 0.3

// 적분항 (비대칭 속도)
if error > 0:
    integralTerm += error × Ki_norm × (감속구간보정)
else:
    integralTerm += error × Ki_norm × 1.5     // 음의 적분은 빠르게 (언더슛 방지)

integralTerm = clamp(integralTerm, minIntegralForLag, maxIntegral)

// 미분항 (양방향)
dTerm = -Kd × smoothedRate

// 축적 에너지 보상 (목표 근처에서만 작동)
storedEnergy = CH1_PV - (목표온도 + 현재열지연)
energyCompensation = -storedEnergy × 0.15 × proximity

// 최종 CH1 SV 계산
newCh1SV = rampedTarget + error × Kp + integralTerm + dTerm + energyCompensation
```

### PI 제어 파라미터

| 변수 | 설명 | 기본값 |
|------|------|--------|
| `Kp` | 비례 게인 | 1.5 (열지연 기반 적응) |
| `Ki_norm` | 정규화 적분 게인 | 0.01 °C/사이클 |
| `Kd` | 미분 게인 | 3.0 (열지연 기반 적응) |
| `feedbackIntervalSec` | PI 제어 주기 | 5초 |

#### 적응형 게인 스케일링

열지연(CH1 PV - 샘플 온도)을 실시간 추적하여 게인을 자동 조정합니다.

```
// 열지연 추적 (리딩 단계: 비대칭 EMA — 증가 시 즉시 반영, 감소 시 서서히)
if currentLag > observedThermalLag and rateOfChange > 0.05:
    observedThermalLag = currentLag           // 즉시 반영
else:
    observedThermalLag = EMA(observedThermalLag, currentLag, α=0.02)

// 게인 스케일링 (기준: 열지연 20°C)
lagScale = clamp(observedThermalLag / 20.0, 0.3, 2.0)

Kp = 1.5 × √lagScale     // 열지연 클수록 → Kp 증가 (느린 시스템 → 빠른 추종)
Kd = 3.0 / √lagScale     // 열지연 클수록 → Kd 감소 (선행 제동 완화)
```

| 열지연 | lagScale | Kp | Kd | 시스템 특성 |
|--------|----------|----|----|-----------|
| 10°C | 0.5 | 1.06 | 4.24 | 빠른 응답 → 오버슈트 방지 강화 |
| 20°C | 1.0 | 1.50 | 3.00 | 기준 |
| 40°C | 2.0 | 2.12 | 2.12 | 느린 응답 → 빠른 추종 강화 |

#### SV 안전 제한

```
// 상한: 램프목표 + 열지연여유 vs 히터 최대온도 중 작은 값
svCeiling = rampedTarget + max(observedThermalLag, 5) × 1.5
upperLimit = min(effectiveMaxTemp, svCeiling)

// ★ BakeoutMaxDeltaT 제한: CH1 SV와 샘플 온도 차이 제한
if BakeoutMaxDeltaT > 0:
    upperLimit = min(upperLimit, 샘플온도 + BakeoutMaxDeltaT)

// 하한: 정상상태에서 필요한 최소 SV (급격한 SV 하락 → 샘플 언더슛 방지)
svLowerBound = max(rampedTarget + observedThermalLag × 0.85,
                   monitorTemp - currentLag × 0.3)

newCh1SV = clamp(newCh1SV, svLowerBound, upperLimit)

// SV 감소 속도 제한 (급격한 하강 방지 — 3단계)
if 샘플온도 < 램프목표:   SV 감소 차단 (아직 올려야 할 구간)
if 샘플온도 ≈ 목표:      최대 0.3°C/사이클 감소 (미세 조정)
if 샘플온도 > 목표:      최대 0.5°C/사이클 감소 (오버슈트 대응)
```

## 3. 온도 유지 (홀드 단계)

### 목표 도달 판정

아래 4가지 조건을 **모두 동시에** `BakeoutStabilizationSeconds`(기본 600초) 동안 **연속 유지**해야 홀드 단계로 진입합니다:

```
조건 1: 모니터온도 ≥ 목표온도 - BakeoutTolerance (기본 1.0°C)
조건 2: 모니터온도 ≤ 목표온도 + BakeoutTolerance
조건 3: 변화율 < 0.3°C/사이클
조건 4: SV 변동 < 0.5°C/사이클 (SV가 아직 변하면 곧 온도도 변할 것)
```

조건 중 하나라도 깨지면 카운터가 리셋되고 다시 처음부터 셉니다.

### TM4 직접 모드 (Ch1만)

- TM4 내장 PID가 자동으로 SV(=목표온도)를 유지
- 소프트웨어는 5초 주기로 안전 체크만 수행

### PI 피드백 모드 (Ch2~12 포함)

승온에서 찾은 적분항(`integralTerm`)을 그대로 유지하며 미세 조정합니다.
이 적분항은 정상상태에서의 히터-샘플 열 지연을 보상하는 역할입니다.

```
holdError = 목표온도 - 모니터온도

// 적분: 대칭 속도로 미세 조정 (홀드에서는 공격적 감쇠 불필요)
integralTerm += holdError × Ki_norm
integralTerm = clamp(integralTerm, 0, maxIntegral)

// D항: 양방향
holdDTerm = -Kd × smoothedRate

// CH1 SV 계산
newCh1 = 목표온도 + holdError × Kp + integralTerm + holdDTerm
newCh1 = clamp(newCh1, 목표온도, effectiveMaxTemp)

// SV 감소 속도 제한 (승온과 동일한 3단계 적용)
```

> 홀드에서는 축적 에너지 보상/감속구간 보정이 없습니다 — 이미 평형 상태이므로 단순 PI로 충분합니다.

## 4. 안전 메커니즘

### 승온/홀드 공통 보호

| 보호 항목 | 감지 조건 | 대응 동작 |
|----------|----------|----------|
| **센서 이상값** | 1회 변화량 > 30°C, 또는 온도 < -10°C | 3회 연속 무시 후 → 안전 종료 |
| **CH1 무응답** | SV > 목표의 50%인데 PV < 목표의 50% | 60초(12회) 연속 → 안전 종료 |
| **통신 끊김** | TempController Modbus 응답 없음 | 30초(6회) 연속 → 안전 종료 |
| **과압** | 챔버 압력 > MaxPressureDuringExperiment | 15초: SV 단계적 감소, 30초: 안전 종료 |
| **CH1 상한 포화** | SV ≥ effectiveMaxTemp이면서 샘플 온도 정체 | 300초(60회) 연속 정체 → 안전 종료 |
| **과온도 워치독** | CH1 PV > effectiveMaxTemp + 20°C | 히터 **즉시** 정지 (별도 워치독 스레드) |
| **SetTemperature 실패** | Modbus 쓰기 응답 없음/에러 | 5회 연속 실패 시 경고 로그 |

### effectiveMaxTemp (모드별 분리)

베이크아웃과 아웃게싱(탈가스율 측정)에서 서로 다른 상한값을 사용합니다:

```
베이크아웃: BakeoutHeaterMaxTemperature (사용자 설정, 기본 150°C)
아웃게싱:   HeaterCh1SetTemperature + 50°C (최소 300°C, TM4 자체 PID가 제어)
```

> 이 분리가 없으면 베이크아웃 150°C 상한이 아웃게싱에도 적용되어, 200°C 아웃게싱 시 170°C(상한+20)에서 과온도 워치독이 히터를 정지시키는 문제가 발생합니다.

## 5. 종료 시퀀스

### BakeoutEndAction별 동작

| 설정 | 동작 | 사용 시나리오 |
|------|------|-------------|
| **HeaterOff** | 전체 종료: 히터 OFF → 냉각 대기 → IG OFF → 터보 정지 → 드라이 정지 → 벤트 | 무인 운전 후 완전 종료 |
| **MaintainTemperature** | 히터/펌프/칠러 현재 상태 유지, AutoRun 프로세스만 완료 | 온도 유지 상태에서 수동 전환 |
| **NotifyOnly** | 모든 장비 상태 유지 + 완료 알림 팝업 | 작업자가 직접 종료 판단 |

### 전체 종료 시퀀스 (HeaterOff) 상세

```
1. 히터 CH1 정지 (Stop 명령)
2. 냉각 대기: CH1 PV ≤ VentingStartTemperature (기본 125°C)까지 대기
   - 이 동안 펌프는 계속 가동 (진공 유지 → 산화 방지)
3. 이온게이지 HV OFF
4. 터보펌프 정지 + 완전 감속 대기 (속도 0 Hz 확인)
5. 게이트밸브 닫기
6. 벤트밸브 + 배기밸브 열기 (대기압 복원)
7. 대기압 도달 대기
8. 드라이펌프 정지
9. 벤트밸브/배기밸브 닫기
```

## 6. 설정 파라미터 요약

| 파라미터 | 단위 | 기본값 | 설명 |
|----------|------|--------|------|
| `BakeoutTargetTemperature` | °C | 100 | 최종 목표 온도 |
| `BakeoutRampRate` | °C/h | 2 | 승온 속도 |
| `BakeoutHoldTimeMinutes` | 분 | 30 | 목표 도달 후 유지 시간 |
| `BakeoutHeaterMaxTemperature` | °C | 150 | CH1 SV 안전 상한 (PI 모드에서 SV가 이 값 초과 불가) |
| `BakeoutMaxDeltaT` | °C | 50 | CH1 SV - 샘플온도 최대 허용 차이 |
| `BakeoutTolerance` | °C | 1.0 | 목표 도달 허용오차 (±) |
| `BakeoutStabilizationSeconds` | 초 | 600 | 안정화 유지 시간 (연속 충족 필요) |
| `BakeoutFeedbackIntervalSec` | 초 | 5 | PI 제어 주기 |
| `BakeoutRiseTimeoutMinutes` | 분 | 0 | 승온 타임아웃 (0=램프속도 기반 자동 계산) |
| `BakeoutEndAction` | enum | HeaterOff | 종료 동작 (HeaterOff / MaintainTemperature / NotifyOnly) |
| `BakeoutMonitorCh1~Ch12` | bool | Ch2만 | 모니터 채널 선택 (다중 선택 시 MAX 기준 제어) |
| `MaxPressureDuringExperiment` | Torr | 1E-4 | 실험 중 최대 허용 압력 |
| `PressureInterlockDurationSeconds` | 초 | 0 | 과압 인터락 지속 시간 (0=기본값: SV감소 15초, 중단 30초) |

## 7. 전체 흐름도

```
AutoRun 시작
  │
  ├─ [단계 1~6] 초기화, 밸브, 펌프, 진공 도달, 이온게이지
  │
  ├─ [단계 7] 히터 시작
  │     │
  │     ├─ 모니터 채널 = Ch1만?
  │     │     │
  │     │     ├─ Yes: TM4 하드웨어 램프 설정 (BakeoutRampRate °C/h)
  │     │     │       TM4 내장 PID가 승온/유지 전담
  │     │     │       소프트웨어: 도달 감시 + 안전 체크만
  │     │     │
  │     │     └─ No:  TM4 하드웨어 램프 OFF
  │     │             소프트웨어 램프(rampedTarget) + PI 피드백으로 Ch1 SV 실시간 제어
  │     │             모니터 채널 중 MAX 온도 기준
  │     │
  │     └─ 칠러 PID: 베이크아웃에서는 비활성화
  │
  ├─ [단계 8-1] 승온 루프 (5초 주기)
  │     ├─ 모니터 온도 읽기 (Modbus)
  │     ├─ 안전 체크 (센서, 과압, 통신, 상한 포화)
  │     ├─ PI 계산 → Ch1 SV 업데이트 (PI 모드만)
  │     └─ 도달 판정: 목표±허용오차 + 안정화 시간?
  │           ├─ No: 계속
  │           └─ Yes: 홀드 진입
  │
  ├─ [단계 8-2] 홀드 루프 (5초 주기, BakeoutHoldTimeMinutes 동안)
  │     ├─ 모니터 온도 읽기
  │     ├─ 안전 체크
  │     ├─ PI 미세 조정 (PI 모드만, 승온의 적분항 유지)
  │     └─ 홀드 시간 경과? → Yes: 종료 시퀀스
  │
  └─ [단계 9] 종료 시퀀스
        ├─ HeaterOff:            히터 OFF → 냉각 → IG OFF → 펌프 정지 → 벤트
        ├─ MaintainTemperature:  현재 상태 유지 (AutoRun만 종료)
        └─ NotifyOnly:           알림만 (모든 상태 유지)
```
